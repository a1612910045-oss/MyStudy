# 异常、调试与资源释放
> 2026-06 | C# · 异常 · 调试

写程序不只是写成功路径。真正可靠的代码，需要认真处理失败路径：参数错误、文件不存在、网络失败、对象为空、资源未释放。

## try / catch

```csharp
try
{
    LoadConfig();
}
catch (FileNotFoundException ex)
{
    Console.WriteLine($"Config missing: {ex.Message}");
}
catch (Exception ex)
{
    Console.WriteLine($"Unexpected error: {ex}");
}
```

捕获异常时，优先捕获具体异常。不要一上来就吞掉所有 `Exception`。

## 什么时候抛异常

异常适合表达“不应该发生的失败”或“调用方必须处理的失败”。

```csharp
public Item GetItem(string id)
{
    if (!items.TryGetValue(id, out var item))
    {
        throw new KeyNotFoundException($"Item not found: {id}");
    }

    return item;
}
```

如果失败是业务常态，可以用 `TryXxx`：

```csharp
public bool TryGetItem(string id, out Item item)
{
    return items.TryGetValue(id, out item);
}
```

## finally

`finally` 无论成功还是失败都会执行：

```csharp
try
{
    Open();
    Work();
}
finally
{
    Close();
}
```

它适合清理资源。

## using

实现 `IDisposable` 的对象可以用 `using` 自动释放：

```csharp
using var stream = File.OpenRead(path);
using var reader = new StreamReader(stream);
string text = reader.ReadToEnd();
```

文件、网络、数据库连接等资源都要注意释放。

## 不要吞异常

危险写法：

```csharp
try
{
    Save();
}
catch
{
}
```

这会让错误消失，调试时非常痛苦。至少记录日志：

```csharp
catch (Exception ex)
{
    logger.LogError(ex);
    throw;
}
```

`throw;` 会保留原始堆栈，`throw ex;` 会重置堆栈，不推荐。

## Debug 与断点

调试时重点看：

- 当前变量值是否符合预期。
- 分支是否走到了你以为的路径。
- 集合数量是否正确。
- 对象是否为 `null`。
- 方法调用顺序是否合理。

Unity 中可以配合：

```csharp
Debug.Log("message");
Debug.LogWarning("warning");
Debug.LogError("error");
```

但不要让日志成为唯一调试方式。断点和调用栈更可靠。

## 参数校验

公共方法应该保护自己的输入：

```csharp
public void AddItem(Item item)
{
    ArgumentNullException.ThrowIfNull(item);
    items.Add(item);
}
```

越靠近边界的代码，越应该做校验。
