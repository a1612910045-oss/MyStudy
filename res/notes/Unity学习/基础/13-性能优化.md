# 性能优化
> 2026-06 | Unity · 性能 · Profiler · GC · DrawCall

性能优化的第一步不是猜哪里慢——是测量。Unity 的 Profiler 是你最重要的优化工具，没有之一。打开它：`Window → Analysis → Profiler`。

## Profiler 怎么看

几个最重要的模块：

| 模块 | 看什么 |
|---|---|
| CPU Usage | 脚本、物理、渲染的 CPU 时间 |
| GPU Usage | 渲染耗时 |
| Memory | 内存占用与 GC 分配 |
| Rendering | DrawCall、三角形数、SetPass Call |

**操作流程**：
1. 运行游戏，让 Profiler 录制一段有代表性的场景（战斗、加载、UI 交互）。
2. 找 CPU 尖峰 → 展开看是哪个函数。
3. 切换到 Memory 模块，看 GC Alloc 峰值——分配越频繁，GC 回收越频繁，帧率波动越大。
4. 切换到 Rendering，看 DrawCall 是否在目标范围内。

## CPU 优化

### 减少 Update 中的开销

Update 每帧都执行。里面的每一行代码都是乘了 60（60fps）的开销。

```csharp
// 不要在 Update 做这些事
void Update()
{
    var player = GameObject.Find("Player");              // Find 遍历整个场景
    var pos = GetComponent<Transform>().position;        // 不需要，直接用 transform
    Debug.Log(transform.position);                       // Log 有巨大开销
    var go = new GameObject();                           // 实例化有分配
}
```

把查找/获取移到 Awake/Start，计算结果缓存起来，日志只在开发期保留。

### 减少字符串操作

```csharp
// 不好：字符串拼接产生 GC
string text = "Score: " + score;

// 好：插值也是拼接，短的不需要太纠结
string text = $"Score: {score}";

// 如果大量连接（UI 文本），用 StringBuilder
var sb = new StringBuilder();
sb.Append("HP: ");
sb.Append(hp);
sb.Append("/");
sb.Append(maxHp);
uiText.text = sb.ToString();
```

### 善用对象池

频繁 Instantiate / Destroy（子弹、敌人、粒子、滚动列表 Item）是 GC 的重灾区：

```csharp
// 对象池的基本思路
public class ObjectPool<T> where T : Component
{
    private Queue<T> pool = new Queue<T>();
    private T prefab;

    public T Get()
    {
        if (pool.Count > 0)
        {
            var obj = pool.Dequeue();
            obj.gameObject.SetActive(true);
            return obj;
        }
        return Instantiate(prefab);
    }

    public void Return(T obj)
    {
        obj.gameObject.SetActive(false);
        pool.Enqueue(obj);
    }
}
```

Unity 2021+ 内置了 `UnityEngine.Pool`：

```csharp
using UnityEngine.Pool;

private ObjectPool<Bullet> bulletPool;

void Start()
{
    bulletPool = new ObjectPool<Bullet>(
        createFunc: () => Instantiate(bulletPrefab),
        actionOnGet: (b) => b.gameObject.SetActive(true),
        actionOnRelease: (b) => b.gameObject.SetActive(false),
        actionOnDestroy: (b) => Destroy(b.gameObject)
    );
}

Bullet bullet = bulletPool.Get();
// 用完
bulletPool.Release(bullet);
```

## GC（垃圾回收）优化

GC 是 Unity 优化的核心词之一。GC 触发时主线程会被暂停，在移动端上可能造成明显卡顿。

**常见 GC 来源**：
- 每帧 new 对象（`new WaitForSeconds`，`new Vector3()`，字符串拼接）。
- 装箱：值类型转 `object`（foreach 循环、string.Format、`Debug.Log(value)`）。
- LINQ：`Where`、`Select`、`OrderBy` 会分配迭代器和临时集合。热路径不要用。
- `GetComponent<T>()` 的某些重载、`FindObjectsOfType<T>()`。
- 协程的 `StartCoroutine(IEnumerator routine)` 本身有分配。

**减少 GC 的技巧**：
- 用 `RaycastNonAlloc` 代替 `RaycastAll`。
- 用 `CompareTag` 代替 `tag == "..."`（tag 的 getter 会分配字符串）。
- 用 `Physics.OverlapSphereNonAlloc` 代替 `OverlapSphere`。
- 在非频繁路径中才使用 LINQ。
- 把 `WaitForSeconds` 缓存到成员变量。

## DrawCall 优化

DrawCall 是 CPU 告诉 GPU「画这个」的次数。每次 DrawCall 都有开销。目标是控制 DrawCall 在合理范围——PC 可以到 1000-2000，移动端尽量 50-200。

**减少 DrawCall 的方式**：
- **静态批处理**：对不动的物体勾选 `Static`（Batching Static），Unity 会将它们合并成一个大的网格。
- **动态批处理**：对小网格（< 300 顶点）的动态物体自动合并。条件严格（相同材质、顶点数限制等）。
- **GPU Instancing**：大量相同模型（草、石头）用同一个材质 + 同一网格，GPU 一次渲染多个。在材质上勾选 `Enable GPU Instancing`。
- **Sprite Atlas**：把多个小图合并成一张大图，用同一材质渲染，减少材质切换（SetPass Call）。
- **Atlas 用于 UI**：UI 中不同 Image 用同一个 Atlas 内的 Sprite，共享材质，减少批次。

## 移动端/微信小游戏特殊优化

- **帧率目标**：移动端通常 30fps 就够（`Application.targetFrameRate = 30`）。60fps 耗电发热严重。
- **纹理压缩**：Android 用 ASTC，iOS 用 ASTC，不要用 RGBA32。
- **关闭垂直同步**：`QualitySettings.vSyncCount = 0`。
- **分辨率适配**：不要用设备原生分辨率渲染。`Screen.SetResolution(1280, 720, true)` 或设置 RenderScale。
- **降低 Shader 复杂度**：微信 WebGL 环境对复杂 Shader 支持有限。避免曲面细分、几何着色器。
- **音频格式**：短音效用 WAV（未压缩），长音频用 MP3/Vorbis 流式加载。微信小游戏对音频格式有限制。

## 优化的正确姿势

1. 先用 Profiler 找到瓶颈（CPU/GPU/Memory/GC）。
2. 针对瓶颈做改动。
3. 再用 Profiler 确认改善了多少。
4. 不要在没测过的情况下凭感觉「优化」。

过早优化是万恶之源。但「知道怎么优化」不是过早——它让你写代码时避开明显的坑，而不是事后花三倍时间重构。
