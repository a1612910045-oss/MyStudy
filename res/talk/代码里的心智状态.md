# 从代码里看程序员的心智状态
> 2026-05 | 杂谈 · 编程哲学 · 代码质量

翻自己半年前写的代码，经常有一种"这真的是我写的吗"的恍惚感。代码会留下痕迹——不只是功能的痕迹，更是写代码时心智状态的痕迹。

## 焦虑型代码

特征：
- 过度防御：每个方法开头都有三层 null 检查，包括不可能为 null 的参数。
- 过度注释：`// 此处将 x 赋值给 y`——生怕自己忘记，实际上只是在翻译代码。
- 类膨胀：一个 PlayerController 里塞了 800 行，因为"改起来方便"。

这通常发生在赶 deadline 或者对项目不熟悉的时期。不知道该信任什么，所以什么都防备。

## 疲惫型代码

特征：
- 命名崩坏：`var a1`, `var a2`, `var temp`——脑子已经转不动命名了。
- 复制粘贴的痕迹：同一段逻辑出现在三个不同文件里，只改了变量名。
- 没有错误处理：`try { ... } catch { }`——空的 catch 块。

通常在深夜或者连续工作 6 小时后出现。这种时候最好的选择是停下。

## 冷静型代码

特征：
- 方法短：平均 15 行以内，一个方法只做一件事。
- 命名自解释：不需要注释也能读懂意图。
- 边界清晰：错误处理明确、资源管理到位、测试顺手写了。

通常出现在思路清楚、没有被打断的上午。

## 为什么要在意这件事

因为代码不是你一个人读的——三个月后的你、接手你代码的同事、被安排来修改这个模块的人，都要面对你留下的心智痕迹。

写得快的代码不一定是好代码。但写的时候心平气和的代码，往往是更好的代码。

下次写代码前，先花两分钟想清楚数据结构。这一小步能避免后面一大片焦虑型代码。
